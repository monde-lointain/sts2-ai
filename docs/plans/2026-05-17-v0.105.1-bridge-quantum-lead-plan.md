## Partitioning strategy (R8 by Q1-target-file family)

### Why Q1-target-file family, not upstream bucket

Every actionable upstream row (PORT or DELETE) resolves to exactly one Q1 file family. Naïve "monsters bucket = stream A; cards bucket = stream B" partitioning fails R8 because **the same Q1 cross-family files (`Phase1Content.cs`, `q4-manifest-phase1.json`, `phase1-silent.json`, `EncounterFixtureTests.cs`) are touched by every bucket** — a `Cards/Abrasive.cs` port adds a card to `Cards/`, registers it in `Phase1Content.cs::BuildCardCatalog`, adds it to the Q1 fixture, and may update the canonical Q4 registry. Parallel streams on cards + powers would collide on `Phase1Content.cs` line-level edits.

Q1-target-file-family partition resolves this by:
1. Drawing the partition boundary around the Q1 file family (Cards, Monsters, Powers, …) so each family owns its leaf files (`Cards/*.cs`, `Cards/Effects/*.cs`).
2. Coalescing cross-family shared files (the catalog Register block, the Q1 fixture, EncounterFixtureTests) into either (a) a per-wave **wave-close commit** owned by one engineer per wave, or (b) per-section locks on the file using marker comments.

This is the strategy used by Wave 3.5 (12 files clean partition: `Phase1Encounters`/`Phase1Monsters`/`Phase1Content` register subset + new slime classes + fixture + tests — all owned by one engineer). Bridge waves replicate the pattern at scale.

### Bucket-vs-substrate mismatch — `scenes-gameplay` + `root-config` filter

Audit of the sidecar against Q1's headless substrate (`engine/headless/`) finds **49 rows the sidecar marks as PORT that have no Q1 file family**:

- **`scenes-gameplay` bucket (47 PORT rows):** all 47 rows are `.tscn` files (Godot scene definitions). Q1's headless engine carries zero `.tscn` files — Q1 is a Godot-free C# library; all scene work belongs to the unported Godot UI layer.
- **`root-config` bucket (2 PORT rows):** `project.godot` + `sts2.csproj` — Godot project descriptor + the upstream's monolithic C# project file. Q1 uses its own `engine/headless/sts2-headless.sln` + per-project `.csproj` files; the upstream `sts2.csproj` is not consumed.

**Recommendation to project-lead:** re-classify these 49 rows as `SURFACE-NO-ACTION` (not `IGNORE` — IGNORE is for buckets like `scenes-ui` / `ui-only` / `multiplayer` that we never want to track; SURFACE-NO-ACTION is "we acknowledge the upstream change exists but Q1 substrate doesn't carry the file"). Mechanism: a Wave 5.6 follow-up tool change to `tools/upstream-sync/src/upstream_sync/diff_analyze.py::assign_decision` that catches `.tscn` + Godot-project-config patterns and reclassifies to SURFACE-NO-ACTION at next `make sync-port-decisions` regeneration.

**Until Wave 5.6 lands:** the bridge plan treats these 49 rows as NO_ACTION_NEEDED in-prose; the JSON sidecar `status` field remains `PENDING` for them until the regeneration; engineer dispatch prompts MUST NOT pull rows from `scenes-gameplay` or `root-config` buckets. Encoded as a forbidden-bucket filter in Wave 6-15 dispatch prompt templates.

This drops the effective bridge actionable count from 869 → **820** (857 PORT − 49 false-positives + 12 DELETE).

### Cross-family shared files (require serialization)

Five Q1 files are touched by **every** content family. Naïve parallelism collides; R8 demands a serialization strategy.

| File | Why shared | Serialization strategy |
|---|---|---|
| `engine/headless/src/Sts2Headless.Domain/Content/Phase1Content.cs` | The Register-block for every catalog (Build*Catalog). Each ported content row adds a `register.Register(X.CanonicalId, new X());` line in the corresponding Build*Catalog method. | **Option A (recommended): per-wave wave-close commit.** Each engineer working a content family in the wave appends their registry block to a file in their worktree (e.g., `engine/headless/.bridge-wave-N-cards.txt` — gitignored). Wave-close engineer (one stream per wave designated "wave-close owner") merges all such fragments into `Phase1Content.cs` in a single final commit after the parallel streams' worktrees merge. Avoids per-line locking; engineers don't see each other's Phase1Content edits. Trade-off: wave-close adds 1-2h serial wall-clock per wave. |
| `engine/headless/test/fixtures/q4-manifest-phase1.json` | Q1 content-coverage gate fixture. Each ported card/monster/power/relic/potion/encounter adds a name to the corresponding array. | Same as Phase1Content.cs — **wave-close commit per wave** appends all family fragments to the appropriate arrays. JSON merge is mechanical (deduplicate + sort + write); no semantic conflict risk. |
| `contracts/registry/phase1-silent.json` | Canonical Q4 registry tokens. New content gets a new token_id per ADR-003 (stability rules: IDs never reused). | **Wave-close commit per wave**, with one engineer responsible for the token_id assignment (monotonic counter from current max). ID assignment is sequential by definition — no parallelism even possible at row level. Trade-off: serializes ID claims; mitigation: pre-assign ID ranges per wave (Wave 7 = IDs 250-300, Wave 8 = 300-350, …) and let each engineer pick within their range. |
| `engine/headless/test/Sts2Headless.Tests.Domain/Content/EncounterFixtureTests.cs` | Encounter-fixture validation test rows. Each new encounter or monster adds rows. | Touched only by **F5 (Encounters)** + **F2 (Monsters)** family changes. F5 + F2 should run in the **same wave** with F5 as wave-close owner (encounters consume monsters); merge order: F2 worktree first, F5 worktree after, F5 wave-close commit appends EncounterFixtureTests rows for both families' deltas. |
| `engine/headless/src/Sts2Headless.Domain/Content/Encounters/Phase1Encounters.cs` | New encounters (Aeonglass renamed-from-Doormaker, etc.) ctor monsters directly. | Touched by **F5 + F2** in tandem. F5 family owns Phase1Encounters.cs; F2 ports the monster classes but does NOT touch Phase1Encounters. If F2 introduces a monster that F5 needs to ctor, the dependency runs F2 → F5 (serial within wave). |

**Decision: Option A (per-wave wave-close commit owned by a designated wave-close stream).** Justification: (a) per-line locking on a 200+ line Phase1Content.cs would require comment markers like `// BEGIN MONSTERS — engineer X` and is error-prone; (b) full serialization of content waves defeats parallelism (8+ content waves × 1 stream each = 8+ days serial vs 3-4 days parallel); (c) Wave 3.5 precedent: a single engineer owning a 12-file cross-family port worked cleanly. Option A applies that precedent at wave scale.

**Wave-close commit cadence:** the wave-close stream is the LAST stream in each wave's parallel set. It's dispatched concurrently with the content streams but instructed to **wait for the other streams' branches to land before starting Phase1Content/fixture/registry edits**. Phase 1 of wave-close stream's work (instrumentation: scan engineer worktrees for `.bridge-wave-N-*.txt` fragments) can begin in parallel; the actual edit happens after parallel merges complete.

## Cross-family shared-file strategy

Recap of the strategy decision above (Option A: per-wave wave-close commit):

- **Each content wave has 3 parallel streams + 1 serial wave-close stream.** Total = 4 streams per wave, but only 3 are concurrent (the 4th waits for the others to land before editing shared files).
- Wave-close stream is dispatched **concurrently** with content streams to keep wall-clock down, but its dispatch prompt instructs: "STOP and wait for all peer streams' branches to be merged to main before beginning Phase1Content/fixture/registry edits. Use that time to (a) inventory the registry-block fragments each peer engineer wrote to their worktree's `.bridge-wave-N-*.txt` file, (b) pre-assign token IDs from the wave's reserved range, (c) draft the Phase1Content + fixture diffs."
- Wave-close stream's branch is merged LAST per wave; sets the wave's pre-tag SHA for `/wave-close`.

Alternative considered + rejected:
- **Option B (per-line locking):** each engineer adds their content to `Phase1Content.cs` with marker comments `// BEGIN-WAVE-N-MONSTERS-α / END-WAVE-N-MONSTERS-α`. Rejected because (i) marker-comment edits would conflict on shared file edits anyway (insertion-order ambiguity); (ii) future spec readers shouldn't see wave-N comments in production code; (iii) Wave 3.5 precedent shows single-engineer-owns-cross-family pattern works fine.
- **Option C (full content-wave serialization):** dispatch one stream per wave; Phase B takes 11+ days serial → ~10-13 waves → ~13 days vs ~7 with parallelism. Rejected because the ~6-day savings is large.
